# Implementation Plan: Flight Training Scheduler

**Branch**: `001-flight-training-scheduler` | **Date**: 2026-04-28 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/001-flight-training-scheduler/spec.md`

## Summary

Implement six Akka SDK components (EventSourcedEntity ×2, View, Consumer, Agent, HttpEndpoint) over pre-existing domain objects (`Timeslot`, `BookingEvent`, `Participant`) to produce a flight training booking system. Domain objects must not be modified. The endpoint orchestrates agent → entity calls because `ComponentClient` is not available in entities.

## Technical Context

**Language/Version**: Java 21 (OpenJDK 21.0.10)
**Primary Dependencies**: Akka SDK 3.5.6 (upgrade to 3.5.18 pending); SLF4J (already in tree)
**Storage**: Akka built-in event journal (in-memory for local dev, replicated in cloud)
**Testing**: JUnit 5 + Akka SDK TestKit (already configured in pom.xml)
**Target Platform**: Local dev (Akka runtime); Akka cloud for deployment
**Project Type**: Event-driven microservice (single service)
**Performance Goals**: Booking confirmed within 5s end-to-end (including agent LLM call)
**Constraints**: No domain object modification; ComponentClient not injectable in entities; free-tier LLM usage
**Scale/Scope**: Certification project — single service, no HA/multi-region requirements

## Constitution Check

### I. Akka SDK First ✅
All six components use Akka SDK primitives exclusively. No custom frameworks, external message brokers, or custom state stores are introduced. The only external dependency addition is the LLM API key (environment variable), not a new Maven dependency.

### II. Design Principles ✅
- **Domain independence**: `Timeslot`, `BookingEvent`, and `Participant` are in `io.example.domain` with no Akka imports. ✅
- **API isolation**: `FlightEndpoint` uses its own `BookingRequest` and `AvailabilityRequest` records, not domain types directly. ✅
- **Single responsibility**: Each of the 6 components has one clear purpose. ✅
- **Descriptive naming**: All components and methods are domain-aligned. ✅

### III. Test Coverage ✅
Each component must have unit or integration tests using the Akka SDK TestKit:
- `BookingSlotEntityTest` — command/event/state transitions
- `ParticipantSlotEntityTest` — command/event/state transitions
- `ParticipantSlotsViewTest` — event-to-row mappings
- `SlotToParticipantConsumerTest` — event dispatch
- `FlightConditionsAgentTest` — mock LLM response
- `FlightEndpointTest` — integration test for each endpoint

### IV. Simplicity ✅
No Workflow introduced (not needed for single synchronous flow). No extra abstractions. YAGNI applies: no generic scheduling framework, no retry infrastructure beyond what Akka provides.

## Implementation Phases

### Phase A: BookingSlotEntity

Implement the core event-sourced entity:

1. **`emptyState()`** — already returns `new Timeslot(HashSet.newHashSet(10), HashSet.newHashSet(10))`. Keep as-is.

2. **`applyEvent(BookingEvent event)`** — switch on all 4 event types:
   - `ParticipantMarkedAvailable` → `currentState().reserve(event)`
   - `ParticipantUnmarkedAvailable` → `currentState().unreserve(event)`
   - `ParticipantBooked` → `currentState().book(event)`
   - `ParticipantCanceled` → `currentState().cancelBooking(event.bookingId())`

3. **`markSlotAvailable(Command.MarkSlotAvailable cmd)`**:
   - Guard: if `available.contains(cmd.participant()) || bookings contains cmd.participant()` → `effects().error(...)`
   - Emit: `effects().persist(new ParticipantMarkedAvailable(entityId, id, type)).thenReply(Done.done())`

4. **`unmarkSlotAvailable(Command.UnmarkSlotAvailable cmd)`**:
   - Emit `ParticipantUnmarkedAvailable` directly (no guard needed; unreserve is safe)
   - `effects().persist(...).thenReply(Done.done())`

5. **`bookSlot(Command.BookReservation cmd)`**:
   - Guard 1: future-slot check — `Instant.parse(entityId).isAfter(Instant.now())` else error
   - Guard 2: `!currentState().isBookable(studentId, aircraftId, instructorId)` → error
   - Emit 3 events (one per participant type):
     ```
     effects().persist(
       new ParticipantBooked(entityId, studentId, STUDENT, bookingId),
       new ParticipantBooked(entityId, aircraftId, AIRCRAFT, bookingId),
       new ParticipantBooked(entityId, instructorId, INSTRUCTOR, bookingId)
     ).thenReply(Done.done())
     ```

6. **`cancelBooking(String bookingId)`**:
   - Guard: `currentState().findBooking(bookingId).isEmpty()` → error
   - Emit 3 `ParticipantCanceled` events for each booking entry found

7. **`getSlot()`**:
   - `effects().reply(currentState())`

---

### Phase B: ParticipantSlotEntity

Implement the per-participant event-sourced entity:

1. **`applyEvent(Event event)`** — switch on all 4 event types:
   - `MarkedAvailable` → `new State(slotId, participantId, participantType, "AVAILABLE")`
   - `UnmarkedAvailable` → `new State(slotId, participantId, participantType, null)`
   - `Booked` → `new State(slotId, participantId, participantType, "BOOKED")`
   - `Canceled` → `new State(slotId, participantId, participantType, null)`

2. **`markAvailable(Commands.MarkAvailable mark)`** → emit `MarkedAvailable` → reply Done
3. **`unmarkAvailable(Commands.UnmarkAvailable unmark)`** → emit `UnmarkedAvailable` → reply Done
4. **`book(Commands.Book book)`** → emit `Booked` → reply Done
5. **`cancel(Commands.Cancel cancel)`** → emit `Canceled` → reply Done

No guards needed — this entity is driven purely by the consumer; it is not called directly by users.

---

### Phase C: ParticipantSlotsView

Implement the read-model view:

1. **`TableUpdater.onEvent(Event event)`** — switch on all 4 event types:
   - `MarkedAvailable` → `effects().updateRow(new SlotRow(slotId, participantId, participantType.name(), null, "AVAILABLE"))`
   - `UnmarkedAvailable` → `effects().deleteRow()`
   - `Booked` → `effects().updateRow(new SlotRow(slotId, participantId, participantType.name(), bookingId, "BOOKED"))`
   - `Canceled` → `effects().deleteRow()`

2. **`getSlotsByParticipant(String participantId)`** — add `@Query`:
   ```java
   @Query("SELECT * FROM slots WHERE participantId = :participantId")
   public QueryEffect<SlotList> getSlotsByParticipant(String participantId)
   ```

3. **`getSlotsByParticipantAndStatus(ParticipantStatusInput input)`** — add `@Query`:
   ```java
   @Query("SELECT * FROM slots WHERE participantId = :participantId AND status = :status")
   public QueryEffect<SlotList> getSlotsByParticipantAndStatus(ParticipantStatusInput input)
   ```

---

### Phase D: SlotToParticipantConsumer

Implement the event consumer:

1. **`onEvent(BookingEvent event)`** — switch on all 4 event types, call the appropriate `ParticipantSlotEntity` command:
   ```java
   case ParticipantMarkedAvailable e -> client
       .forEventSourcedEntity(participantSlotId(event))
       .method(ParticipantSlotEntity::markAvailable)
       .invoke(new Commands.MarkAvailable(e.slotId(), e.participantId(), e.participantType()));
   // ... similar for other 3 events
   ```
2. Each case returns `effects().done()` after the call.
3. The `participantSlotId` helper is already implemented in the stub — use it as-is.

---

### Phase E: FlightConditionsAgent

Implement the AI agent:

1. **`SYSTEM_MESSAGE`** — clear system prompt instructing the agent to:
   - Call `getWeatherForecast(timeSlotId)` to get conditions
   - Evaluate: visibility ≥ 5 miles, wind ≤ 25 knots, no precipitation
   - Return a `ConditionsReport` with `meetsRequirements` (true/false) and a brief reason

2. **`query(String timeSlotId)`** — update the `userMessage` to include the slotId and request structured JSON output:
   ```java
   effects()
     .systemMessage(SYSTEM_MESSAGE)
     .userMessage("Evaluate flight conditions for time slot: " + timeSlotId)
     .responseAs(ConditionsReport.class)
     .thenReply()
   ```

3. **`getWeatherForecast(String timeSlotId)`** — implement deterministic mock:
   - Returns bad conditions string if `timeSlotId.contains("storm")` or `timeSlotId.contains("bad")`
   - Returns good conditions string otherwise
   - This avoids external HTTP calls and stays within free tier

---

### Phase F: FlightEndpoint

Implement the HTTP endpoint (all methods already stubbed):

1. **`markAvailable(String slotId, AvailabilityRequest request)`**:
   - Parse `participantType` (already done in stub with try/catch)
   - Call: `componentClient.forEventSourcedEntity(slotId).method(BookingSlotEntity::markSlotAvailable).invoke(new MarkSlotAvailable(new Participant(request.participantId(), participantType)))`
   - Return `HttpResponses.ok()`

2. **`unmarkAvailable(String slotId, AvailabilityRequest request)`**:
   - Same pattern but call `BookingSlotEntity::unmarkSlotAvailable`

3. **`getSlot(String slotId)`**:
   - Call `BookingSlotEntity::getSlot`, return the `Timeslot`

4. **`createBooking(String slotId, BookingRequest request)`** (orchestration):
   ```java
   // Step 1: check conditions
   var report = componentClient.forAgent()
       .inSession(slotId)
       .method(FlightConditionsAgent::query)
       .invoke(slotId);
   if (!report.meetsRequirements()) {
       throw HttpException.badRequest("Flight conditions unacceptable");
   }
   // Step 2: book the entity
   componentClient.forEventSourcedEntity(slotId)
       .method(BookingSlotEntity::bookSlot)
       .invoke(new BookReservation(request.studentId(), request.aircraftId(), request.instructorId(), request.bookingId()));
   return HttpResponses.created();
   ```

5. **`cancelBooking(String slotId, String bookingId)`**:
   - Call `BookingSlotEntity::cancelBooking` with `bookingId`
   - Return `HttpResponses.ok()`

6. **`slotsByStatus(String participantId, String status)`**:
   - Call `ParticipantSlotsView::getSlotsByParticipantAndStatus` with `new ParticipantStatusInput(participantId, status.toUpperCase())`
   - Return the `SlotList`
